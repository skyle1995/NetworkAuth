## 一、运营模式（计费核心）

`operation_mode` 决定账号「可用性」以什么为依据，是所有计费逻辑的总开关。

| 模式 | 值 | 可用性依据 | 说明 |
|---|---|---|---|
| **免费模式** | `2` | 恒可用 | 不计费。账号**即便已到期 / 无点数也可正常登录使用**；心跳照常校验但不扣费；转绑不扣费。已充值的时长/点数**保留但不消耗** |
| 时长模式 | `0` | `expired_at` 未到期 | 按到期时间。到期即不可登录/掉线 |
| 点数模式 | `1` | `points` 余额 | 按点数余额，细分「按次 / 按时」两种扣费 |

> ⚠️ **模式决定一切「时长 / 点数」字段的单位**：同一个「扣除」配置，时长模式扣**分钟**，点数模式扣**点数**，免费模式**不扣**。

### 1.1 免费模式（`operation_mode=2`）

- 登录：不校验到期/余额，**过期账号也能登录**。
- 心跳（`41`）：照常校验令牌与账号状态（封停/拉黑仍拦截），但**不扣任何费用、不因到期拒绝**。
- 转绑（`51`）：**一律不扣费**，仍照常计次、执行换绑。
- 功能扣点（`53`）：返回「当前应用非点数模式」。
- 客户端应**忽略 `expired_at` / `points`**，视为始终可用。
- 「例外」：之前已充值激活的时长/点数字段值**原样保留**，只是登录/心跳不再看它、也不消耗——将来切回时长/点数模式即恢复按原值计费。

### 1.2 时长模式（`operation_mode=0`）

- 账号 `expired_at` 未到期即可用；`permanent=true`（到期=2099 年）为永久，永不到期、扣时无效。
- 登录/心跳发现已到期 → 返回「账号已到期」。
- 充值（`22`）、领试用（`24`）在当前到期时间基础上**顺延分钟数**。

### 1.3 点数模式（`operation_mode=1`）

`points_charge_mode` 决定扣费方式：

#### 按次（`points_charge_mode=0`）
- 每次登录扣 `points_per_login` 点；**`0` = 登录不扣**（点数仅由 `53 功能扣点` 显式消耗）。
- 余额不足登录被拒。登录成功后会话内不再以点数拦截。

#### 按时（`points_charge_mode=1`）
- 每 `points_period_minutes` 分钟扣 `points_per_period` 点（周期最小 1 分钟、每周期扣点最小 1）。
- 结算时机由 `points_heartbeat_charge`（「扣费触发」）决定：

  | 扣费触发 | 值 | 登录时 | 心跳时 |
  |---|---|---|---|
  | 登录预扣 | `0` | 立即扣 1 个周期并顺延 | 到期后自动续扣下一周期 |
  | 心跳触发 | `1` | **不扣** | 首个周期延到首次心跳时扣 |

- **心跳默认按周期扣费**；客户端对「免费功能」的心跳传 `no_charge:true` 可**跳过本次扣费**（点数不变、仍可用）。以此实现「功能A免费 / 功能B计费」。
  - ⚠️ **心跳扣费语义已反转**（相较旧版）：旧字段 `charge`（默认不扣、传 `true` 才扣）已改为 `no_charge`（默认扣、传 `true` 才不扣）。升级后**客户端必须同步改**，否则计费错乱。仅影响「点数-按时」模式。
- 离线不补扣：跨过多个周期只按「当前时间」重新起算一个周期，不追扣欠费。

## 二、登录与多开设置

| 设置 | 字段 | 取值 | 逻辑 |
|---|---|---|---|
| 登录方式 | `login_type` | 0 顶号 / 1 非顶号 | 多开满员时：**顶号**=踢掉最早的一个「开」放新登录进来；**非顶号**=拒绝新登录（返回「已达最大同时在线数」） |
| 多开范围 | `multi_open_scope` | 0 单设备 / 1 单IP / 2 全部设备 | 决定「一个开」以什么为单位计数：按机器码 / 按登录IP / 按会话（每次登录都算独立一开） |
| 多开数量 | `multi_open_count` | ≥1 | 允许同时存在的不同「开」的数量。⚠️ **`≤0` 会被后端兜底为 1，不是无限** |
| 清理间隔 | `clean_interval` | 小时（默认1） | 后台清理任务对**该应用**的运行周期（基础节拍 1 分钟，按此值决定隔多久真正扫一次失效会话） |
| 心跳间隔 | `check_interval` | 分钟（默认10） | 登录后随 `heartbeat_interval` 返回给客户端，告知其多久心跳一次（不配置回退 10） |
| 自动离线 | `offline_timeout` | 分钟（默认30） | 超过该时长未心跳 → 会话判失效（登录时清理 + 后台扫描删除；不配置回退 30） |

> ⚠️ **三个时间字段关系**：后端强制 `自动离线 ≥ 心跳间隔`，否则客户端还没来得及心跳就被判离线。
> `多开范围=全部设备` 时，同一台机器多次登录也各算一开——想「一机一开」应选 `单设备`。

## 三、机器码 / IP 绑定与验证

### 3.1 验证开关

| 设置 | 字段 | 取值 | 逻辑 |
|---|---|---|---|
| 机器码验证 | `machine_verify` | 0 关 / 1 开 | 开启后登录校验设备码：已绑放行；未绑且未超多开数则**自动绑定**；超出则提示「请先机器码转绑」 |
| IP地址验证 | `ip_verify` | 0 关 / 1 精确 / 2 同市 / 3 同省 | 首次登录**自动绑当前 IP**；后续按级别判定（精确=IP 相同，市/省=归属地相同）。市/省级依赖 IP 归属地库 |

> ⚠️ 机器码验证仅在 `machine_verify=1` 且客户端**传了 `machine_code`** 时才生效；`ip_verify=0` 时完全不绑 IP。

### 3.2 转绑（重绑）设置

机器码与 IP 各有一组，字段前缀分别为 `machine_` / `ip_`：

| 设置 | 字段 | 取值 | 逻辑 |
|---|---|---|---|
| 重绑开关 | `*_rebind_enabled` | 0 关 / 1 开 | 关闭时调转绑接口报「未开启转绑」。两维度都开则依次转绑、各自独立计次/扣费 |
| 重绑限制 | `*_rebind_limit` | 0 每天 / 1 永久 | **每天**=跨天重置计数（免费次数每天恢复）；**永久**=累计不重置 |
| 免费次数 | `*_free_count` | ≥0 | 前 N 次转绑**不扣费** |
| 重绑次数 | `*_rebind_count` | ≥0 | 转绑**次数上限**。⚠️ **`0` = 不限次数** |
| 重绑扣除 | `*_rebind_deduct` | ≥0 | 超免费次数后每次扣除的量（时长模式扣分钟、点数模式扣点数、**免费模式不扣**） |

**单次转绑的判定链**（`used` = 该账号已用次数）：

```
used < 免费次数              → 免费放行
免费次数 ≤ used < 重绑次数    → 扣 deduct
（重绑次数=0 视为无上限）
used ≥ 重绑次数（且 >0）       → 拒绝「转绑次数已达上限」
```

> ⚠️ **重绑次数=0 ≠ 不扣费**：不限次数下，超过免费次数后每次仍按 deduct 扣费。要「完全免费无限转绑」需 `重绑次数=0` 且（`重绑扣除=0` 或 运营为免费模式）。
> **防误重绑**：新值 = 当前已绑值时直接放行，**不计次、不扣费**（幂等，避免同设备白扣）。
> 转绑接口**凭账号密码鉴权、不需登录令牌**——设备/IP 对不上、登不进的用户也能转绑（避免死循环）。

## 四、账号注册设置

| 设置 | 字段 | 取值 | 逻辑 |
|---|---|---|---|
| 账号注册 | `register_enabled` | 0 关 / 1 开 | 关闭则注册接口直接拒绝 |
| 邮箱验证 | `email_verify_enabled` | 0 关 / 1 开 | 开启须先发验证码（`23`）、注册带 `code`。依赖系统 SMTP + Redis |
| 卡密注册 | `card_register_enabled` | 0 关 / 1 开 | 开启后注册须额外提交**有效卡密**；注册即核销该卡并按运营模式发放卡面值（时长发到期时长、点数发点数） |
| IP限制 | `register_limit_enabled` | 0 关 / 1 开 | 按注册 IP 统计，达上限拒绝 |
| 设备限制 | `register_device_limit_enabled` | 0 关 / 1 开 | ⚠️ 开启后**注册必须带 `machine_code`**，按注册设备统计；换 IP 绕不过 |
| 限制时间 | `register_limit_time` | 0 每天 / 1 永久 | **IP 与设备两维度共用** |
| 注册次数 | `register_count` | ≥1 | IP、设备两维度**共用同一上限值** |

> ⚠️ 注册成功**不下发 token**；无试用/卡密时账号**初始即过期**，需再登录/充值/领试用才能用。
> IP 限制与设备限制可单开或同开，但「限制时间 / 注册次数」是**两者共享**的同一组参数。

## 五、领取试用设置

领取试用是**独立接口（`24`）**，不是注册自动赠送。

| 设置 | 字段 | 取值 | 逻辑 |
|---|---|---|---|
| 领取试用 | `trial_enabled` | 0 关 / 1 开 | 关闭或时长≤0 时报「未开启试用领取」 |
| 限制时间 | `trial_limit_time` | 0 每天 / 1 永久 | 每天=每天可领一次；永久=每账号仅一次 |
| 领取方案 | `trial_claim_mode` | 0 无限制 / 1 到期可领 | **无限制**=满足次数即可领；**到期可领**（默认）=仅时长已到期 / 点数为 0 才可领，免费模式不可领 |
| 试用时长/点数 | `trial_duration` | ≥1 | **时长模式=分钟数**、**点数模式=点数**；在当前到期/余额基础上叠加 |

## 七、基础信息与更新策略

| 设置 | 字段 | 取值 | 逻辑 |
|---|---|---|---|
| 应用名称 | `name` | 文本 | 必填 |
| 应用版本 | `version` | 文本（默认 1.0.0） | 客户端 `3 检测最新版本` 据此比对 |
| 应用状态 | `status` | 0 禁用 / 1 启用 | 禁用后该应用所有接口不可用 |
| 更新方式 | `download_type` | 0 不启用 / 1 强制更新 / 2 自由更新 | **强制更新**=版本过旧则**直接拒绝登录**（返回 `code=2` + 更新信息，不发令牌）；**自由更新**=登录成功但提示可更新，用户可不更新；`0` 不下发更新。非 0 时须配 `download_url`。开启后**登录须提交客户端 `version`**；**心跳不做版本判断**，避免打断使用 |
| 下载地址 | `download_url` | URL | 更新方式非 0 时展示/使用 |
| 应用数据 | `app_data` | 文本（base64） | 下发给所有用户的公共数据（`42` 读取），单独入口编辑 |
| 程序公告 | `announcement` | 文本（base64） | 随 `1 获取程序公告` 下发，单独入口编辑 |

> ⚠️ **更新方式已合并**：旧版「强制更新开关(`force_update`) + 更新方式(自动/手动)」合并为单一 `download_type` 三态，不再区分自动/手动。`force_update` 字段**已彻底移除**（模型、数据库列、客户端返回都不再有）；客户端改用 `download_type`（=1 强制、=2 自由）判断。存量数据在服务启动时自动规整（强制→1、其余启用→2），并删除废弃的 `force_update` 列。

## 九、常见坑点汇总

1. **「转绑次数=0」是不限次数，但不等于不扣费**——超过免费次数后仍按扣除额扣时长/点数。
2. **「多开数量=0」不是无限**——后端兜底为 1；要放开多开请填具体数量。
3. **免费模式下 `expired_at` / `points` 仍存在但不生效**——客户端别拿它判断可用性，一律视为可用。
4. **心跳扣费参数从 `charge` 反转为 `no_charge`**——升级后旧客户端若还传 `charge`，点数-按时模式会计费错乱（默认变成扣费）。
5. **设备注册限制开启后注册必须带 `machine_code`**，否则直接被拒；且 IP/设备限制共用「限制时间 / 注册次数」。
6. **自动离线时长要 ≥ 心跳间隔**，否则用户正常在线也会被判掉线。
7. **注册不自动发时长**——无卡密/试用时新账号初始即过期，属正常，需再充值/领试用/登录。
8. **模式切换的单位差异**——把应用从时长模式切到点数模式后，所有「扣除/试用/发放」配置的含义都从「分钟」变为「点数」，需重新核对数值。
